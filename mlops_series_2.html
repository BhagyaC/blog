Introduction to Containers

 

Inorder to build apps we used its own physical machines in the earlier cases, and on top of that we have the kernel and dependencies and followed by the code for the application itself. If we had to scale up the model we have to find more machines to add on. The time for each of these took a lot of time. Hence came the concept of virtualisation where we can implement multiple operating system in the same physical computer. In earlier case each physical computer was for serving a single purpose. Hypervisor is the layer it breaks the dependencies of the operating system to the underlying hardware so that anything can be run on anyone. It improved the deployment also the portability, virtualisation can be converted into images and can be transferred between systems. There is an issue with virtualisation on the single operating system. The dependencies used by different applications are shared it will cause a starving situation. Also, dependency upgrading in one resource might cause issue in another. We cab run the integration tests to make sure that it works, but most of the time dependency failures are hard to find. The simple solution is having dedicated VMs for each application, kernels are separate and they won’t be affect. But the kernels are copied and os update will cause issue in a large scale(booting up the kernal will take a lot of time too). To address this, we only need the user spaces as the virtual machines. Here comes the concept of containers. Containers are the dedicated and isolated user spaces to running application. We package the entire dependencies in containers and they run on any kernel. This idea leads to the concept of microservices loosely coupled modular design patterns that allows the os to scale and upgrade the components without affecting the whole..
